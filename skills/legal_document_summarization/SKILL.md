# 法律摘要

## 概述表格

| 项目 | 内容 |
|------|------|
| **能力编号** | 15 |
| **能力名称** | 法律摘要 |
| **能力类型** | 信息提取与结构化 |
| **核心功能** | 对各类法律文书进行类型识别、要素提取、逻辑梳理和结构化输出 |
| **输入** | 判决书、裁定书、调解书、仲裁裁决书、行政处罚决定书等法律文书全文 |
| **输出** | 按文书类型定制的结构化摘要（案件信息、事实、焦点、依据、结论、说理） |
| **关联能力** | 法律文档格式化（确保输出格式规范）、法条检索（核实法律依据的准确性）、法律推理（分析说理逻辑） |
| **关键原则** | 忠实原文、客观中立、突出核心、区分类型 |

## 法律声明

> **重要提示：**
> 1. 摘要必须忠实于原文，不得添加原文未载明的事实、观点或结论。
> 2. 摘要中不得出现主观评价性措辞（如"法院英明""处罚合理""被告恶意"等）。
> 3. 对于扫描件或图片形式的文书，OCR 识别可能存在误差，摘要中应标注不确定性。
> 4. 摘要结论仅反映该文书所载内容，不代表对案件实体问题的法律意见。
> 5. 涉及正在审理中的案件，摘要不得泄露不应公开的程序信息或商业秘密。

## 一、核心概念

### 1.1 法律文书类型与摘要重点对照

不同类型的法律文书，其核心功能和信息结构不同，摘要重点必须随之调整：

| 文书类型 | 核心功能 | 摘要重点 | 区别于其他类型的关键要素 |
|----------|----------|----------|------------------------|
| **判决书** | 解决实体争议 | 事实认定、法律适用、裁判理由、判决结果 | 法院对事实的"认定"（非当事人主张） |
| **裁定书** | 解决程序事项 | 程序背景、申请/异议内容、裁定依据、程序结论 | 不涉及实体争议，仅处理程序问题 |
| **调解书** | 确认双方合意 | 争议事项、调解协议内容、履行安排 | 体现双方自愿，非法院强制裁判 |
| **仲裁裁决书** | 仲裁庭终局裁决 | 仲裁请求、事实认定、仲裁庭意见、裁决结果 | 一裁终局、仲裁协议基础 |
| **行政处罚决定书** | 行政制裁 | 违法事实、处罚依据、处罚种类和幅度 | 行政机关单方作出，可复议可诉讼 |

### 1.2 摘要 vs 缩写 vs 评价

```
原文 ──────────────────────────────────────→

  摘要：提取核心要素，结构化重组，省略非关键细节
  
  缩写：按比例压缩原文，保留所有信息点的精简版
  
  评价：加入主观判断和价值评论（摘要中禁止）
```

**摘要的本质：** 不是简单缩短原文，而是识别核心信息、剔除无关细节、按逻辑结构重组信息。

### 1.3 六要素提取框架

所有法律文书摘要均应围绕以下六要素展开：

```
┌─────────────────────────────────────────────────┐
│  1. 案件信息：谁、何时、何地、何机构作出          │
│  2. 事实背景：发生了什么（与结论相关的事实）       │
│  3. 争议焦点：需要解决的核心法律问题              │
│  4. 法律依据：援引了哪些规范                      │
│  5. 结论结果：最终如何处理                        │
│  6. 说理过程：如何从事实+规范推导出结论           │
└─────────────────────────────────────────────────┘
```

## 二、完整工作流程

### 阶段一：准备与识别

#### 步骤 1：识别文书类型

首先判断该法律文书属于哪一种类型：

| 判断依据 | 判决书 | 裁定书 | 调解书 | 仲裁裁决书 | 行政处罚决定书 |
|----------|--------|--------|--------|-----------|---------------|
| **标题关键词** | "民事判决书""刑事判决书""行政判决书" | "民事裁定书""执行裁定书" | "民事调解书""调解协议" | "仲裁裁决书" | "行政处罚决定书" |
| **核心功能** | 实体裁判 | 程序处理 | 确认合意 | 仲裁裁决 | 行政制裁 |
| **作出主体** | 法院 | 法院 | 法院（确认双方合意） | 仲裁委员会/仲裁庭 | 行政机关 |

**判断原则：** 不能先套格式再识别文书，否则容易遗漏该类文书的真正重点。

#### 步骤 2：提取案件信息

从文书首部或标题部分提取完整引用信息：

| 提取项 | 来源位置 | 示例 |
|--------|----------|------|
| **案件名称** | 文书首部 | 美景公司诉蚂蚁公司不正当竞争纠纷案 |
| **案号/文号** | 文书标题下方 | （2024）京0108民初1234号 |
| **日期** | 文书末尾 | 2024年12月7日 |
| **作出机构** | 文书首部/末尾 | 北京市海淀区人民法院 |
| **当事人** | 文书首部 | 原告：美景公司；被告：蚂蚁公司 |

**要求：** 该部分应尽量保持原文表述，以便检索与引用。

### 阶段二：内容提取

#### 步骤 3：提炼关键事实或程序背景

根据文书类型，提炼与结论直接相关的事实或程序事项：

**判决书/仲裁裁决书/行政处罚决定书：**
- 提炼实体争议相关事实
- 区分"当事人主张的事实"与"文书认定的事实"
- 仅保留影响结论的关键事实，剔除无关细节

**裁定书：**
- 提炼程序背景（如：原告起诉后被告提出管辖权异议）
- 提炼申请事项或异议内容
- 不涉及实体争议的事实

**调解书：**
- 提炼争议事项与达成调解的背景
- 双方的主要分歧点
- 促成调解的关键因素（如有提及）

#### 步骤 4：归纳争议焦点

将文书中的争议提炼为一个或多个明确的法律问题：

**要求：**
- 表述为法律问题（可回答的疑问句）
- 抓住裁判真正解决的问题
- 避免写成过于宽泛的背景描述

| 差的争议焦点 | 好的争议焦点 |
|-------------|-------------|
| "双方发生纠纷" | "美景公司爬取蚂蚁平台数据的行为是否构成不正当竞争" |
| "存在争议" | "被告提出的管辖权异议是否成立" |
| "赔偿问题" | "原告主张的500万元损害赔偿数额是否有事实和法律依据" |

#### 步骤 5：提取结论或处理结果

准确概括文书最终的处理结论：

| 文书类型 | 结论表述方式 | 示例 |
|----------|-------------|------|
| **判决书** | 判决支持/驳回/确认/变更 | "判决被告停止涉案不正当竞争行为，赔偿原告经济损失200万元" |
| **裁定书** | 裁定准许/驳回/移送/中止/终结 | "裁定驳回被告对本案管辖权提出的异议" |
| **调解书** | 双方达成如下协议 | "双方自愿达成如下调解协议：一、被告于X月X日前支付原告XX元..." |
| **仲裁裁决书** | 裁决... | "裁决被申请人向申请人支付货款人民币100万元及逾期利息" |
| **行政处罚决定书** | 决定给予... | "决定责令当事人改正违法行为，并处罚款人民币50万元" |

**关键原则：** 准确区分"当事人主张"和"文书结论"。不能把原告主张误写成法院判决。

#### 步骤 6：提取法律依据

列出文书援引的法律规范：

- **法律**：《民法典》第X条、《刑法》第X条
- **司法解释**：最高法关于...的解释第X条
- **行政法规/部门规章**
- **合同条款**（如涉及合同纠纷）
- **仲裁规则**（仲裁裁决书）
- **行政裁量基准**（行政处罚决定书）

**注意事项：**
- 优先提取被文书直接援引并作为裁判/裁决/处罚依据的规范
- 如文书引用的法条已失效或被修订，可在摘要中标注
- 对于"参照""参考"的规范，可单独列示

#### 步骤 7：概括说理过程

用简洁语言概述作出结论的逻辑过程：

**说理过程应展示：**
1. 文书采纳了哪些事实（证据采信情况）
2. 如何适用法律规范（法条解释和适用）
3. 如何回应主要争议与抗辩（对反对意见的回应）
4. 如何从事实+规范推导出结论（逻辑链条）

**要求：**
- 不是简单缩写原文，而是展示"事实如何被认定→规范如何被适用→结论如何得出"
- 语言精炼，突出因果链条
- 避免复述完整的诉辩经过

### 阶段三：输出

#### 步骤 8：按文书类型格式化输出

根据识别的文书类型，选择对应的结构化模板输出（详见第三部分的输出格式模板）。

## 四、示例

### 示例 1：民事判决书摘要

**输入（原文要点）：**
> 北京市海淀区人民法院民事判决书（2024）京0108民初1234号。原告美景公司诉被告蚂蚁公司不正当竞争纠纷。原告主张被告蚂蚁公司通过技术手段抓取原告运营的"XX数据平台"上的数据，用于自身产品运营，构成不正当竞争。被告辩称其抓取的是公开数据，且已采取合理措施，不构成不正当竞争。法院认定：被告未经许可抓取原告平台数据，该数据虽部分公开但需投入大量成本收集整理，被告行为实质性地替代了原告数据产品，损害了原告合法权益，扰乱市场竞争秩序。判决：一、被告立即停止涉案不正当竞争行为；二、被告赔偿原告经济损失200万元及合理开支10万元。

**输出：**

```markdown
## 判决书摘要

### 一、案件信息
- **案件名称：** 美景公司诉蚂蚁公司不正当竞争纠纷案
- **案号：** （2024）京0108民初1234号
- **法院：** 北京市海淀区人民法院
- **裁判日期：** [原文日期]

### 二、案件事实
原告美景公司运营"XX数据平台"，投入大量成本收集整理数据。被告蚂蚁公司通过技术手段抓取原告平台数据用于自身产品运营。原告起诉主张被告行为构成不正当竞争。

### 三、争议焦点
1. 被告抓取原告平台数据的行为是否构成不正当竞争？
2. 若构成不正当竞争，损害赔偿数额如何确定？

### 四、法院认定事实
- 被告未经许可抓取原告平台数据
- 涉案数据虽部分公开，但需投入大量成本收集整理
- 被告行为实质性地替代了原告数据产品
- 被告行为损害了原告合法权益，扰乱市场竞争秩序

### 五、法律依据
- 《反不正当竞争法》第2条（一般条款）
- 《反不正当竞争法》第12条（互联网专条）

### 六、裁判结果
1. 被告立即停止涉案不正当竞争行为
2. 被告赔偿原告经济损失200万元
3. 被告赔偿原告合理开支10万元

### 七、说理过程
法院认定原告对其平台数据享有竞争性权益，被告抓取行为未经许可以且实质替代原告数据产品，违反了诚实信用原则和公认的商业道德，扰乱市场竞争秩序，构成不正当竞争。关于赔偿数额，法院综合考虑原告数据产品的市场价值、被告侵权行为的性质和持续时间等因素，酌定赔偿200万元。
```

### 示例 2：裁定书摘要

**输入（原文要点）：**
> 北京市第一中级人民法院民事裁定书。原告美景公司诉被告蚂蚁公司不正当竞争纠纷，向海淀区法院起诉。被告在答辩期内提出管辖权异议，主张本案应由被告住所地法院（北京市朝阳区）管辖。海淀区法院审查认为，本案系不正当竞争纠纷，侵权行为地（海淀区）法院具有管辖权，裁定驳回被告的管辖权异议。被告不服，提起上诉。一中院审查认为，原告主张的侵权行为发生在海淀区，海淀区法院作为侵权行为地法院对本案有管辖权。裁定：驳回上诉，维持原裁定。

**输出：**

```markdown
## 裁定书摘要

### 一、案件信息
- **案件名称：** 美景公司诉蚂蚁公司不正当竞争纠纷案（管辖权异议上诉）
- **案号：** [案号]
- **法院：** 北京市第一中级人民法院
- **裁定日期：** [日期]

### 二、程序背景
原告向海淀区法院起诉，被告提出管辖权异议，一审法院裁定驳回。被告不服，提起上诉。

### 三、申请或异议内容
上诉人（原审被告）主张本案应由被告住所地法院（朝阳区法院）管辖。

### 四、争议焦点
本案应由被告住所地法院管辖，还是由侵权行为地法院管辖？

### 五、法律依据
- 《民事诉讼法》第29条（侵权行为地管辖）
- 《民事诉讼法》第130条（管辖权异议审查）

### 六、裁定结果
裁定驳回上诉，维持原裁定（海淀区法院对本案有管辖权）。

### 七、说理过程
二审法院审查认为，本案系不正当竞争纠纷，原告主张的侵权行为发生在海淀区。根据《民事诉讼法》第29条，因侵权行为提起的诉讼，由侵权行为地或被告住所地法院管辖。海淀区作为侵权行为地，其法院对本案具有管辖权。一审裁定正确，应予维持。
```

### 示例 3：行政处罚决定书摘要

**输入（原文要点）：**
> 北京市市场监督管理局行政处罚决定书。当事人：某科技有限公司。经查，当事人在其运营的App中，未经用户同意收集用户剪贴板信息，用于识别用户来源渠道。该行为违反《个人信息保护法》第13条、第16条。当事人申辩称其收集信息用于优化用户体验，且已采取匿名化处理。行政机关审查认为，剪贴板信息属于个人信息，收集前未取得用户同意，违反最小必要原则，匿名化抗辩不能成立。处罚决定：责令改正违法行为，处罚款50万元。

**输出：**

```markdown
## 行政处罚决定书摘要

### 一、案件信息
- **处罚决定书文号：** [文号]
- **作出机关：** 北京市市场监督管理局
- **日期：** [日期]

### 二、当事人信息
- **被处罚人：** 某科技有限公司
- **统一社会信用代码：** [...]

### 三、违法事实
当事人在其运营的App中，未经用户同意收集用户剪贴板信息，用于识别用户来源渠道。

### 四、争议焦点
1. 剪贴板信息是否属于应受保护的个人信息？
2. 当事人以"优化用户体验"和"匿名化处理"为由的抗辩是否成立？

### 五、法律依据
- **违法认定依据：** 《个人信息保护法》第13条（处理个人信息应当取得个人同意）、第16条（不得以个人不同意处理其个人信息或撤回同意为由拒绝提供产品或服务）
- **处罚依据：** 《个人信息保护法》第66条

### 六、处罚结果
- 责令改正违法行为
- 罚款人民币50万元

### 七、说理过程
行政机关认定：剪贴板信息能够识别特定自然人，属于个人信息。当事人在收集前未征得用户同意，且该收集行为超出实现产品功能所必需的范围，违反最小必要原则。当事人关于"优化用户体验"的目的不能豁免同意义务，"匿名化处理"抗辩与事实不符。综合考虑违法行为的性质、情节和危害程度，决定处罚款50万元。

### 八、救济途径
- **行政复议：** 收到决定书之日起60日内向[复议机关]申请
- **行政诉讼：** 收到决定书之日起6个月内向[管辖法院]提起
```

## 五、常见错误与防范

| 错误类型 | 错误描述 | 后果 | 防范措施 |
|----------|----------|------|----------|
| **类型识别错误** | 将裁定书按判决书格式摘要，大篇幅提取实体争议 | 摘要重点完全偏离，遗漏真正的程序性焦点 | 首先识别文书类型，再选择对应模板 |
| **主张与认定混淆** | 将原告主张的事实写成"法院认定的事实" | 摘要失实，误导读者 | 严格区分"当事人主张"和"文书认定"，后者须标注来源 |
| **争议焦点过于宽泛** | "双方发生纠纷""存在争议" | 无法揭示案件真正需要解决的法律问题 | 将焦点表述为可回答的法律问题 |
| **主观评价混入** | "法院英明地认为""处罚十分合理" | 违反摘要的客观中立原则 | 删除所有评价性形容词，仅陈述事实和结论 |
| **复述而非摘要** | 大段引用原文，仅做简单删减 | 失去摘要的价值，篇幅冗长 | 提取核心要素后用概括性语言重组 |
| **遗漏法律依据** | 仅写"根据相关法律规定" | 无法追溯裁判/处罚的规范基础 | 逐条列出文书援引的具体法条 |
| **说理过程空洞** | "法院依法作出判决" | 未展示逻辑链条，读者无法理解结论如何得出 | 展示"事实认定→规范适用→结论推导"的完整链条 |
| **忽视程序性套话** | 大量保留送达经过、开庭通知、合议庭组成 | 篇幅膨胀，核心信息被淹没 | 压缩或省略对结论无影响的程序性内容 |

## 六、特殊场景处理

### 6.1 多被告/多原告案件

- 在"当事人"部分清晰列明所有当事人的诉讼地位
- 在"案件事实"中区分各当事人的不同主张和行为
- 在"裁判结果"中分别列明对各当事人的处理

### 6.2 系列案件/关联案件

- 在摘要开头标注本案与关联案件的关系
- 如本案判决结果依赖于另案判决，应予以说明
- 如涉及批量诉讼（如证券虚假陈述系列案），可简化共同事实，突出个案差异

### 6.3 二审/再审文书

- **二审判决书摘要：** 重点提取一审认定与二审改判/维持的差异点，二审法院的改判理由
- **再审文书摘要：** 重点提取当事人申请再审的理由、法院对原审的审查结论
- 在"程序背景"中说明案件经历的审级

### 6.4 扫描件/OCR 识别文书

- 如存在识别不确定的文字，在摘要中用`[？]`标注
- 关键数字（金额、日期、案号）如识别不清，应特别标注
- 建议用户核对原文，尤其是法律依据的条文序号

### 6.5 涉外/港澳台案件

- 注意识别适用的法律（中国法、外国法、国际条约）
- 如涉及法律冲突规范，应说明法院选择的准据法
- 涉及域外送达、域外取证等程序问题的，在裁定书中重点提取

### 6.6 集团诉讼/代表人诉讼

- 在"当事人"部分说明诉讼代表人和被代表的原告群体
- 在"案件事实"中概括共同的事实基础
- 在"裁判结果"中说明判决的效力范围

## 七、质量检查清单

```markdown
□ 1. 文书类型是否已准确识别（判决/裁定/调解/仲裁/行政处罚）？
□ 2. 案件信息是否完整（名称、案号、日期、作出机构、当事人）？
□ 3. 事实部分是否仅保留了与结论直接相关的信息？
□ 4. 是否严格区分了"当事人主张"和"文书认定"？
□ 5. 争议焦点是否已提炼为明确的法律问题？
□ 6. 结论是否准确反映了文书的最终处理结果？
□ 7. 法律依据是否已列出具体法条（法律名称+条文序号）？
□ 8. 说理过程是否展示了"事实→规范→结论"的逻辑链条？
□ 9. 摘要中是否未混入任何主观评价性措辞？
□ 10. 输出格式是否与文书类型匹配（使用了正确的模板）？
□ 11. 程序性套话是否已适度压缩？
□ 12. 扫描件/OCR 识别的不确定之处是否已标注？
```

## 九、局限性与风险提示

- **信息完整性依赖：** 摘要质量完全依赖原文信息的完整性和清晰度。原文逻辑混乱或说理不充分的，摘要无法弥补。
- **OCR 识别风险：** 扫描件的 OCR 识别可能存在文字、数字、标点错误，涉及关键信息时应建议用户核对原文。
- **主观理解偏差：** 尽管摘要要求客观中立，但对"关键事实"的选择和"争议焦点"的提炼仍不可避免地涉及判断，不同人可能有不同侧重。
- **格式限制：** 标准化模板无法覆盖所有特殊类型文书，遇到新型或不常见文书时，应在通用六要素框架基础上灵活调整。
- **时效性：** 法律依据的时效性以文书作出时点为准，摘要中引用的法条可能已被后续修订，如需用于当前法律分析，应另行核实。
- **保密信息：** 涉及未公开审理、商业秘密、个人隐私的案件，摘要应遵守信息披露限制，必要时对敏感信息进行脱敏处理。
