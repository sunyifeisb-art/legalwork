# LegalWork AI 系统实施项目计划

> 基于《三大对标律所AI系统具体真实功能清单》的整合实施计划
> 参照对象：植德律所（私有RAG）、大成昆明律所（法院节点对标）、澜湄+惠畅律所（诉讼标准化中台）
> 创建日期：2026-06-13 | 版本：draft v1

---

## 一、总体架构

### 1.1 项目愿景

将 LegalWork 从"文档OCR+脱敏能力包"升级为覆盖**案件全生命周期**的律所AI中台系统，融合三家标杆律所AI能力：

```
┌─────────────────────────────────────────────────────────────┐
│                   LegalWork AI 系统                          │
├─────────────────┬─────────────────┬─────────────────────────┤
│  植德模式        │  大成昆明模式    │  澜湄+惠畅模式          │
│  (RAG知识库)    │  (法院节点对标)   │  (诉讼标准化中台)        │
├─────────────────┼─────────────────┼─────────────────────────┤
│ · 私有知识库    │ · 立案AI预审    │ · 要素标准化拆解        │
│ · AI律师助理    │ · 85节点预警    │ · 成果文件自动生成      │
│ · 类案检索      │ · 卷宗AI阅卷    │ · 争议焦点预判          │
│ · 文书初稿      │ · 多模型适配    │ · 庭审提纲生成          │
│ · 案件复盘      │ · 数据安全封闭  │ · 质量标准化质控        │
└─────────────────┴─────────────────┴─────────────────────────┘
```

### 1.3 差距分析（对标三家律所）

| 标杆特征 | 差异项 | 当前状态 | 优先级 |
|---------|--------|---------|-------|
| **植德: 私有RAG知识库+团队经验沉淀** | 已有基础检索，缺少团队经验闭环（胜诉/败诉自动复盘归档） | case_system有knowledge模块但无复盘流程 | P0 |
| **植德: 分业务垂直AI助理矩阵** | 已有skills机制，缺少基于场景的助理路由和UI交互界面 | skills目录60+但无场景化组合调度 | P1 |
| **大成: 立案AI预审自检模拟** | 缺少立案材料校验引擎、法院格式模板库 | 完全缺失 | P0 |
| **大成: 85审判节点预警督办** | 仅有CourtAIPredictor(6节点)，需扩展至85节点+自动预警 | 6/85节点 | P0 |
| **大成: 多模型分环节适配** | 缺少模型路由层，目前单一规则预测 | 完全缺失 | P1 |
| **大成: 全本地加密存储** | 已有脱敏+本地存储，但无司法合规级别的加密存储策略 | 部分完成 | P1 |
| **澜湄: 诉讼要素标准化拆解** | 已有要素提取能力，缺少案由维度的标准化模板和全自动填充 | 部分完成 | P0 |
| **澜湄: 全套成果文件自动产出** | DocumentDrafter仅有骨架，缺少完整内容生成链路 | 骨架完成 | P0 |
| **澜湄: 争议焦点AI归纳** | CourtAIPredictor覆盖了部分，缺少独立焦点归纳服务 | 部分完成 | P1 |
| **澜湄: 质量标准化质控** | 缺少文书校对、法条引用校验、逻辑瑕疵检测 | 完全缺失 | P1 |
| **澜湄: 办案经验AI沉淀固化** | 缺少新人的"经验复制"机制 | 完全缺失 | P2 |
| **通用: 团队协作中台** | 单机版，缺少多人协作、角色权限、工作流引擎 | 完全缺失 | P2 |

## 二、实施路线图（3个里程碑）

### 里程碑一：基础能力补齐（P0） — 约4周

> 目标：补齐对标三家律所的核心缺失能力，让系统可在实际案件中使用

#### M1.1 立案AI预审自检引擎（对标大成昆明）

**实现路径**：
1. 新建 `case_system/court_filing/` 模块
   - `precheck.py` — 立案材料预审引擎（核心类 FilingPrecheckEngine）
   - `templates/` — 法院立案格式模板库（基于民事/行政/刑事公开标准）
   - `rules/` — 案由校验规则（基于《民事案件案由规定》结构化）
2. 核心功能：
   - 材料完整性校验：起诉状、证据清单、授权委托书等必备文件清单检查
   - 格式校验：命名规范、页码、签名、日期格式、排版要求
   - 案由校验：匹配《民事案件案由规定》层级树，自动建议正确案由
   - 管辖校验：地域管辖、级别管辖、专属管辖规则引擎
   - 输出：预审校验报告（通过/不通过+问题列表+修改建议）
3. 新建 skill：`skills/court_filing_precheck/SKILL.md` — 立案预审技能指令
4. Flask API 新增蓝图：`/api/cases/<matter_id>/filing/precheck`
5. CaseAIOrchestrator 注册预审能力

**验收条件**：对给定案件材料可输出结构化预审校验报告，包含材料清单、格式问题、案由建议和管辖分析。

**参考对标**：大成昆明"立案AI预审自检模拟——材料校验、格式校验、案由校验，一次通过不被退回"

---

#### M1.2 85审判节点预警系统（对标大成昆明）

**实现路径**：
1. 新建 `case_system/court_nodes/` 模块
   - `nodes.py` — 85审判节点定义（基于民事诉讼法/司法解释结构化）
   - `scheduler.py` — 节点时间线计算器（起诉→立案→举证→开庭→判决→上诉→执行）
   - `watcher.py` — 节点预警督办（设置预警阈值：7天/3天/1天/过期）
2. 节点分类：
   - 诉前阶段（立案受理、诉前保全、诉前调解等）
   - 审理阶段（答辩期、举证期、证据交换、庭前会议、开庭等）
   - 判决阶段（合议、宣判、文书送达等）
   - 上诉阶段（上诉期、二审立案、二审开庭等）
   - 执行阶段（申请执行、执行查控、评估拍卖等）
3. 首期覆盖：民商事诉讼Top30高频节点，后期扩展至85节点
4. Flask API 新增蓝图：`/api/cases/<matter_id>/court-nodes`
5. 时间线基于案件关键日期自动推算

**验收条件**：对给定案件可自动生成85节点时间线，显示当前进度和下一个关键节点，节点到期前自动预警。

**参考对标**：大成昆明"同步法院85个审判节点——举证期、答辩期、送达、上诉期自动倒计时预警"

---

#### M1.3 私有RAG知识库增强（对标植德）

**实现路径**：
1. 增强现有 `case_system/core.py` 中 KnowledgeBase 类
   - 新增 `case_system/knowledge/` 模块（从core.py拆分）
   - `rag_engine.py` — 私有RAG问答引擎（基于本地向量检索+上下文组装）
   - `experience.py` — 团队经验沉淀闭环（胜诉/败诉复盘自动归档）
   - `chunking.py` — 智能分块策略（法律文书专用分块：按条款、按事实、按法律依据）
2. 关键能力：
   - 基于已有向量检索的问答接口
   - 结案复盘自动归档（从案件材料+任务产出中提炼经验入库）
   - 知识库文件变更自动同步（watchdog监听）
   - 敏感数据分级脱敏（复用redaction模块）
3. 专用提示词模板：`prompts/rag_qa.md` — RAG问答系统提示词
4. API 增强：`/api/cases/<matter_id>/knowledge/ask` — RAG问答接口
5. API 增强：`/api/cases/<matter_id>/knowledge/review` — 案件复盘自动归档

**验收条件**：可对新案件材料进行RAG问答检索；结案后可自动提取经验要点归档；知识库文件变更自动增量同步。

**参考对标**：植德"所内历年文书、判例、办案笔记AI秒查" + "胜诉败诉案件自动总结经验，入库固化团队方法论"

---

#### M1.4 诉讼标准化产出（对标澜湄+惠畅）

**实现路径**：
1. 新建 `case_system/standardization/` 模块
   - `templates/` — 诉讼要素模板库（按案由分类）
     - `civil/` — 民事案由模板（合同纠纷、侵权纠纷、物权纠纷等）
     - `commercial/` — 商事案由模板（公司纠纷、票据纠纷等）
   - `extractor.py` — 要素自动提取引擎（从案件材料中按模板字段自动填充）
   - `generator.py` — 成果文件生成器（立案自检表、要素核对表、代理词、质证意见等）
   - `focus_engine.py` — 争议焦点归纳引擎
   - `trial_outline.py` — 庭审提纲生成器
2. 成果文件类型：
   - 立案自检表（L01）
   - 要素核对表（L02）
   - 类案检索报告（L03）
   - 类案区分意见书（L04）
   - 代理词初稿（L05）
   - 质证意见初稿（L06）
   - 庭审辩论提纲（L07）
   - 庭审质证提纲（L08）
3. 首期覆盖：Top5高频民事案由（买卖合同、民间借贷、劳动争议、交通事故、离婚纠纷）
4. Flask API 新增蓝图：`/api/cases/<matter_id>/standardization`

**验收条件**：对Top5案由的案件材料，可自动输出立案自检表、要素核对表、代理词初稿、质证意见初稿，争议焦点归纳准确率≥70%。

**参考对标**：澜湄+惠畅"固定诉讼办案要素模板，AI一键填全案核心事实" + "立案自检表、要素核对表、类案检索报告、代理词初稿自动生成"

---

#### M1.5 质量标准化质控引擎（对标澜湄+惠畅）

**实现路径**：
1. 新建 `case_system/quality/` 模块
   - `checker.py` — 文书质量检查器
     - 字词校验（错别字、标点符号、数字格式）
     - 逻辑校验（前后矛盾、因果断裂、循环论证）
     - 格式校验（标题层级、段落编号、引用格式）
   - `law_citation.py` — 法条引用校验器
     - 法条格式规范性检查
     - 法条有效性验证（基于内置法律规范库）
     - 废弃/修改法条提醒
   - `report.py` — 质控报告生成器
2. 多种检查模式：
   - 快速检查（字词+格式，毫秒级）
   - 深度检查（逻辑+法条，需文本分析）
3. Flask API 新增蓝图：`/api/cases/<matter_id>/quality/check`

**验收条件**：对给定文书可输出质控报告，含字词问题、格式问题、逻辑问题和法条引用问题，可标记严重等级。

**参考对标**：澜湄+惠畅"文书错漏、逻辑瑕疵、法条引用错误AI自动校对"

### 里程碑二：场景化AI助理与可视化（P1） — 约3周

> 目标：为用户提供直观的AI交互界面和法院预判可视化

#### M2.1 分业务垂直AI助理矩阵（对标植德）

**实现路径**：
1. 新建 `case_system/assistants/` 模块
   - `registry.py` — AI助理注册中心（skills与助理的映射）
   - `router.py` — 场景化路由（根据案件类型、任务自动推荐助理）
   - `orchestrator.py` — 多助理协同编排
2. 预设助理清单：
   - 诉讼案件助理 — 要素提取、文书初稿、类案检索
   - 立案预审助理 — 材料校验、案由建议、管辖分析
   - 庭审准备助理 — 争议焦点归纳、庭审提纲、质证要点
   - 案件复盘助理 — 经验总结、知识归档
   - 合同审查助理 — 合同风险审查（复用skills/contract_risk_review）
3. 助理推荐算法：
   - 基于案由 → 推荐对应助理
   - 基于材料类型 → 推荐处理流程
   - 基于用户查询 → 语义匹配最佳助理
4. Flask API 新增蓝图：`/api/assistants`
5. Desktop 前端集成：助理选择面板

**验收条件**：用户可从桌面端按案件类型选择AI助理，助理自动加载对应skills和工作流。

**参考对标**：植德"不做大平台，只做律师办案高频刚需AI助理矩阵" + "分业务垂直AI助理"

---

#### M2.2 法院AI预判可视化（对标大成昆明）

**实现路径**：
1. 增强 `CourtAIPredictor`（现有6节点→扩展至15+节点）
   - 新增节点：案由判定、诉讼时效、主体适格、管辖权异议、证据三性等
   - 置信度基于文本信号+要素匹配度联合计算
   - 本地规则可配置（支持自定义权重）
2. 新增可视化输出：
   - 节点概率雷达图数据
   - 各节点"支撑材料"索引（直接关联到案件材料片段）
   - 预判报告JSON结构优化（机读+人读双格式）
3. 流程节点适配建议：
   - 法院AI怎么审 → 律所AI就怎么防
   - 每个节点附带"对冲建议"（应补充什么材料、应准备什么论点）
4. Flask API 增强：`/api/court-ai/predict-v2`

**验收条件**：预判报告含15+节点、每个节点带置信度和材料来源索引、附带对冲建议。

**参考对标**：大成昆明"法院AI怎么审，律所AI就怎么防" + "提前预判法院AI会怎么归纳事实"

---

#### M2.3 争议焦点AI归纳（对标澜湄+惠畅）

**实现路径**：
1. 在 `case_system/standardization/` 中实现焦点归纳引擎
2. 核心能力：
   - 从诉状/答辩状/证据中自动提取双方争议要点
   - 按"事实争议"、"法律争议"、"程序争议"分类
   - 归纳法院可能采用的焦点表述方式
   - 预判法院AI归纳焦点，提前做好对冲准备
3. 可用已有 skills：`dispute_issue_identification`、`analogical_reasoning`

**验收条件**：对100页以下案件材料可输出争议焦点清单，分类合理并与法院AI预判兼容。

**参考对标**：澜湄+惠畅"提前预判法院AI归纳焦点，提前做好对冲准备"

---

### 里程碑三：团队协作与批量办案（P2） — 约3-4周

> 目标：支持多人协作、角色权限、批量办案，实现中台化运营

#### M3.1 AI+年轻律师中台作业（对标澜湄+惠畅）

**实现路径**：
1. 在 `case_system/` 中新增工作流引擎：
   - 角色定义：合伙人（终审）、中台律师（修改）、年轻律师（初稿+AI辅助）
   - 任务分配：AI出初稿→中台做基础修改→合伙人终审
   - 状态流转：初稿→复核→终审→定稿
2. 工作区新增状态管理：
   - 案件概览看板（所有案件按状态分组）
   - 个人任务列表（待办/进行中/已完成）
3. 桌面端新增视图：
   - 任务看板
   - 案件列表+筛选

**验收条件**：支持3种角色、任务可分配和流转、案件看板可查看所有案件状态。

**参考对标**：澜湄+惠畅"AI出初稿、中台做基础修改、合伙人只终审决策"

---

#### M3.2 经验沉淀与新人复制（对标澜湄+惠畅+植德）

**实现路径**：
1. 增强案件复盘功能：
   - 裁判结果自动分析（胜诉/败诉原因提取）
   - 律师策略总结（什么论点有效、什么证据关键）
   - 经验标签化入库（按案由、争议类型、法院层级打标）
2. 新人引导：
   - 按案由推荐历史相似案件经验
   - 自动生成"案件处理指南"（基于历史复盘+模板）
   - 新人作业质量自动比较（与历史标杆比对）

**验收条件**：结案后自动生成复盘报告并入库；新人接手新案时自动推送相关经验。

**参考对标**：澜湄+惠畅"老律师办案经验AI沉淀固化，新人无需手把手带教，批量复制办案能力"

---

#### M3.3 批量办案中台（对标澜湄+惠畅）

**实现路径**：
1. 在 `case_system/` 中新增批量处理管线：
   - 批量案件创建（从Excel/CSV导入案件列表）
   - 批量材料导入（对应案件自动关联）
   - 批量要素提取
   - 批量成果文件生成
2. 统计看板：
   - 案件处理效率（人/案/天）
   - 质量指标（质控通过率、返工率）
   - 团队工作量统计

**验收条件**：可同时导入10+同类案件，批量完成要素提取和成果文件生成。

## 九、架构设计原则

1. **渐进增强**：新模块不破坏现有功能，以插件方式注册到 `CaseAIOrchestrator`
2. **数据隔离**：每个案件独立工作区（已有 CaseStore 机制），新模块沿用相同模式
3. **配置驱动**：节点规则、案由目录、要素模板均使用YAML/JSON配置，支持热更新
4. **可测试**：每个模块提供纯函数核心逻辑，便于单元测试
5. **复用优先**：优先使用已有 skills/ 目录中的60+技能，避免重复实现
6. **API优先**：所有能力通过 Flask REST API 暴露，前端可渐进迭代
