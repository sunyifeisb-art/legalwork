# 技术与数据规则（高价值补充）

下面内容是从外部建议中筛出的**值得保留、可落地、应优先执行**的部分；其中属于未来扩展的内容，按“后续评估”处理，不视为已定方案。

## 1. 先做小样本验证，不先追求完美数据库
优先用小样本验证核心假设，例如：
- 少量核心法规
- 少量真实隐私政策

验证重点：
- 能否定位关键条款（如第三方共享、跨境传输）
- 能否关联到对应法规要求
- 能否减少漏判和误报
- 能否输出可溯源的结构化结论

建议默认记录四类评价维度：召回、准确、可溯源、可读性。

## 2. 法规材料优先结构化
推荐：
- Markdown / 纯文本优于 PDF / Word
- 每份法规文件增加头部元数据，例如法规名称、发布机关、效力层级、适用场景、相关条款

## 3. 技术链路优先“三段式”
默认拆成：
1. 检索
2. 判断
3. 报告生成

这样更便于调试，也更容易定位问题出在召回、判断还是输出。

第一阶段如果结果不好，优先先分辨：
- 是没找到对的条文
- 还是找到了但判断错
- 还是判断对了但报告表达差

## 4. 资料分类优先按合规逻辑，不按部门
优先按数据生命周期或场景分类，例如：
- 收集
- 存储
- 使用
- 出境
- 删除

数据出境场景可单独做专题包。

## 5. 法律类 AI 必做三项控制
### A. 分段检索
法规长文本不要整篇直接塞给模型，应切分后只送相关段落。

### B. 引用溯源
每个风险结论尽量带具体法规依据，避免无来源判断。

### C. 人工审核节点
高风险判断默认预留人工确认，不把模型输出当最终法律结论。

## 6. 应提前识别的真实难点
- 法规更新会导致知识库过期
- 隐私政策格式不统一，前期应限制输入或增加清洗
- 合规判断很多是程度题，适合做风险等级而不是简单二元结论
- 长文本与上下文窗口天然冲突，必须靠分段检索缓解
- 法律术语接近但不等同，检索阶段需要关键词/元数据/重排序协同
- 工作流平台本身可能存在批量读取、外部连接、状态持久化限制
- OCR / 扫描件解析应放到后续阶段，不宜一开始做重

## 7. 当前更稳的项目结论
- MVP 先只做“隐私政策数据出境条款审查”是合理的
- 产品定位应始终是“辅助合规效率工具”，不是替代律师或法务作最终判断
- 界面和报告里应保留免责声明与人工复核意识
