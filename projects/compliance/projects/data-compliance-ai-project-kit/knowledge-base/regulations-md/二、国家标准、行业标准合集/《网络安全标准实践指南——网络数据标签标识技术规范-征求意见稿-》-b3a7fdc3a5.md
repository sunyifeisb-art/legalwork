国数据安全法》《中华人民共和国个人信息保护法》《网
络数据安全管理条例》等法律法规，帮助数据处理者使用

本实践指南通过规定网络数据标签标识技术的术语和
定义、属性格式、生成规则、绑定和传输规则、接收和核
验规则、日志要求、安全要求等内容，实现网络数据标签
标识技术标准化，可用于帮助数据处理者对数据进行标签
标识，在此基础上重点对重要数据和个人信息进行分类分
级保护，加强数据全周期全过程溯源管理。

5.2 网络数据标签标识技术应用场景

网络数据标签标识技术适用于网络数据处理者组织内

注：规模化数据流动包括单次批量传输和多次单条累积传输两种情形。
4
数据在网络数据处理者组织内部存储使用时，网络数据
标签标识多用于内部访问控制等数据管理需要，可标记数据

态场景。

网络数据标签标识技术应用全生命周期包括数据标签

6 网络数据标签标识属性格式和生成规则
6.1 静态场景下网络数据标签标识的属性格式

6.1.1 数据源主体
数据源主体属性字段表示该数据产生的源头主体，采用
GB 32100 中规定的法人和其他组织统一社会信用代码编码
的方式，占位 18 个字节。
6.1.2 数据提供方
数据提供方属性字段表示该数据的直接提供方，采用法
人和其他组织统一社会信用代码编码的方式，占位 18 个字
节。


6
6.1.3 分类分级
数据分类分级属性字段占位 1 个字节，采用二进制编码
的方式，按照 GB/T 43697 中相应规则或行业主管部门认定
情况对数据对象进行判定是否属于核心数据、重要数据、一
般数据、个人信息、敏感个人信息，分别依次对应 8 个比特
位的前 5 位，第 6 位表示后续数据出境情况，第 7、8 位作
为扩展位留作以后备用，比特位为“1”标识属于对应级别数据，
比特位为“0”标识不属于对应级别数据。
6.2 动态场景下网络数据标签标识的属性格式

6.2.1 数据标签标识码
数据标签标识码字段用于表征本数据标签的唯一性，由
数据发送方结合时间标识、机器硬件标识、数据属性散列值
等数据按照 GB/T 17969.8 规定的国际通用算法规则 UUID 进
行编码。占位 32 字节。
6.2.2 数据源主体
数据源主体属性字段表示该数据采集或产生的源头主
体，采用 GB 32100 中规定的法人和其他组织统一社会信用
代码编码的方式。占位 18 字节。
6.2.3 数据发送方
数据发送方属性字段表示本次数据传输中该数据的发

7
送主体，采用 GB 32100 中规定的法人和其他组织统一社会
信用代码编码的方式，占位 18 个字节。
6.2.4 数据接收方
数据接收方属性字段表示本次数据传输中该数据的接
收主体，采用 GB 32100 中规定的法人和其他组织统一社会
信用代码编码的方式。占位 18 字节。
6.2.5 分类分级
该字段与静态数据标签标识中的分类分级属性字段相
对应，反映数据的敏感程度与重要性，便于分级保护。占位
1 字节。
6.2.6 数据规模
数据规模属性字段表示本次传输中本类数据的传输数
量，单位为“条”，采用十进制数字字符表示，用于核查数
据流动是否超出规定阈值。占位 16 字节。
6.2.7 时间戳
时间戳属性字段由两部分组成，起始时间和时间间隔。
第一部分标记数据发送传输的起始时间，精确到毫秒，占位
16 字节。第二部分标记时间间隔，描述所生成的动态数据标
签标识的有效期限，精确到秒，占位 8 字节。总占位 24 字
节。
注：如果是单条数据传输该部分设置为空；如果涉及多条数据传输，按照
8
7.2.3 中注 1 设定。

6.2.8 数据传输校验码
对所传输数据进行哈希运算，哈希运算值作为标签的校
验字段随标签发送给数据接收方，用于校验数据的完整性并
实现标签与数据的一一对应。占位 32 字节。
6.2.9 扩展字段
在上述字段基础上，网络数据处理者组织还可根据实际
业务需要增加扩展字段，扩展字段可包含行业分类、安全控
制要求、数据传输用途和留存期限等内容，本指南不再对扩
展字段进行规定。

7.1.1 非结构/半结构化数据：嵌入式绑定

打标，应符合以下要求：
a）单个文件：非结构化/半结构化数据一般以文件形式
存在，宜在文件头部属性或元数据中嵌入网络数据标签标识

10
信息。
示例：

诸如 doc、pdf、cvs 等文本类文件，图片、音频、视频等流文件，在文件头

b）文件集合：在对文件进行分级管理基础上，宜将同

结构化数据一般以数据库形式存在，数据处理者在不改

方式实现；数据处理者新建数据库系统，也可采用直接增加
数据源主体、数据提供方、分类分级信息字段的方式实现。

7.2 动态场景网络数据标签标识的传输

7.2.1 非结构/半结构化数据：嵌入式传输
嵌入式传输方式，即网络数据标签标识嵌入到数据对象
进行绑定，在传输时随着数据对象一起传输。
示例：

非结构化/半结构化数据一般以文件形式存在，通常有 FTP/HTTP/P2P/SMTP

11
等多种传输形式，宜将动态网络数据标签标识（文件）与待传输文件或文件集合
打包成一个文件夹进行传输。

7.2.2 结构化数据：同步传输
数据发送方以同步传输形式进行动态场景下数据打标，
应符合以下要求：
同步传输方式，即在发送方将数据发送给接收方同时，
将生成的动态网络数据标签标识封装在应用层协议头部随
数据同步传输的方式，数据发送方自动化生成并封装网络数
据标签标识，数据接收方自动化解析提取识别网络数据标签
标识。网络数据标签标识通过封装的方式和数据对象进行绑
定。
示例：

结构化数据动态流转一般采用 API 接口方式进行传输，数据传输应用层协
议大多采用 HTTPS 协议，具体可参考附录 B.4 所示。其他协议可参考之，如 FTP
协议可通过控制通道自定义命令实现。
以 HTTP 为例，具体实现方式是在系统统一的 HTTP 调用模块中加入标签注
入逻辑，由程序在每次发起数据请求前自动生成唯一的标签标识，并在 HTTP 协
议头部新增相应字段（如网络数据标签标识码、数据分类分级信息、时间戳等）。
所有调用将继续通过 TLS 加密通道传输，确保标签与数据同步加密。由于跨主
体的系统可能使用不同开发语言和接口框架（如 Java、C++、Python 等），因此
需分别修改各系统的接口封装层（负责发起请求、封装参数、处理响应等，需要
在发送方中增加标签注入逻辑并保证标签的一致性等，在接收方同理）。在数据
接收方系统中，需要在网络数据标签标识接收环节增加一个拦截或过滤组件，用
于识别、解析请求头中的标签字段，并将标签信息与请求元数据（如数据分类分
级信息、时间戳等）一并记录到日志或数据库中。具体实现方式可参考附录 C。

7.2.3 结构化数据：异步传输
数据发送方以异步传输形式进行动态场景下的数据打
12
标，应符合以下要求：
a）异步传输方式，就是网络数据标签标识和数据本身有
时序传输，即在该模式下按照数据发送方和数据接收方约定，
网络数据标签标识在数据传输开始前或者完成后再单独传
输，该方式不需要在业务系统应用层中加入动态标签。
b）网络数据标签标识和数据对象通过网络数据标签标
识的“数据传输校验”字段进行逻辑绑定，该字段通过对原始
数据做哈希运算进行设置，实际应用中可由数据发送方的接
口功能实现。
c）对于频繁多次传输结构类型相同的单条数据（可参考
附录 A.2 销售服务场景），可对批量数据进行统一数据打标，
具体通过设定时间周期或设定计数器两种方式实现，时间周
期和计数器的设定可以根据实际需要由数据发送方和接收
方约定适当的粒度。
注 1：设定时间周期模式（数据传输完成后再传输数据标签）。先设定标签
生成周期（如 1 天即 86400 秒，则在数据标签时间戳字段中的时间间
隔设置为“00086400”）；从该周期内发送第一条数据的起始时间开始
算起（在数据标签时间戳字段中的起始时间设置为该时刻），统计在
已设定标签生成周期内向同一个应用场景下的同一个第三方实体发送
的数据条数（在数据标签数据规模字段设置为该数量）；设置最后一
条数据的哈希值作为数据标签传输校验字段；当该时间周期内的最后
一条数据传输完毕后再发送数据标签。
注 2：设定计数器模式（数据传输前传输数据标签）。先设定计数器，即标
签生成累计条数（如 1000 条，即 1000 条数据累计生成一个数据标签，
数据标签数据规模字段设置为“0000000000001000”）；发送时间设置
为第一条数据拟发送时间；设置第一条数据的哈希值作为数据标签传
输校验字段。

7.2.4 协作平台：中转传输
对于重要数据处理者、一定规模以上的个人信息处理者
等重点网络数据处理者，可选择适配协作平台方式来实现网
络数据标签标识的存档、核验等。

7.2.4.1 网络数据标签标识的存档

数据发送方在发送数据前，将生成的网络数据动态标签
标识存档至协作平台，之后将相应的数据标识发送给数据接
收方，传输方式可以选择本文件 7.2.2 中规定的同步传输方
式或者 7.2.3 中规定的异步传输方式，数据打标位置可以选
择应用层或者网络层。网络数据标签标识和数据对象的绑定
通过存档实现。
注 1：数据标识的生成可以有两种方式：在数据发送方生成的网络数据动态
标签标识中截取网络数据标签码字段，作为数据标识使用；由协作平
台将数据发送方发来的网络数据动态标签标识映射为一个 32 字节的散
列值作为数据标识返回给数据发送方，作为数据标识使用。
注 2：应用层打标可以参考 2.2.2 中规定的方式。
注 3：当数据发送方和数据接收方均采用 Ipv6 进行网络传输时可以选择在
网络层打标，具体可以写入 IPv6 报文扩展头部。

7.2.4.2 网络数据标签标识的核验

数据接收方在收到所传输的网络数据和相应的数据标
识后，向协作平台发起核验请求；协作平台根据存档信息对
网络数据标签标识的完整性、真实性进行核查，并返回核查
结果。
14
7.2.4.3 日志留存和安全要求

涉及日志留存，应按照本文件第 9 章相关要求执行；涉
及涉及安全防护，应按照本文件第 10 章相关安全要求执行。

8 网络数据标签标识的验标规则
8.1 网络数据标签标识接收

数据接收方收到发送方发送的数据对象和相应动态数
据标签标识后，将动态数据标签标识转换为静态数据标签标
识进行本地识别和存储，应根据接收数据的数据结构类型采
用不同的处理方式：
a）对于非结构化/半结构化数据，将动态标签标识中的
源主体、发送方、分类分级字段取出作为静态标签，参考非
结构化/半结构化数据的静态标签打标方式，在文件属性中嵌
入相应的静态标签数据，也可以将多个文件归类并在文件夹
中增加静态标签。
b）对于结构化数据将动态标签标识中的源主体、发送
方、分类分级字段取出作为静态标签，参考结构化数据的静
态标签打标方式，在标签标识映射表中增加或者修改标识映
射信息。
8.2 网络数据标签标识核验

数据接收方应将接收到的数据标签与数据进行一致性
15
核验。
8.3 网络数据标签标识更新

当数据对象属性发生改变（如敏感个人信息已脱敏处理、
个人信息累计量达到重要数据标准等），与数据标签标识内
容不再匹配时，数据接收方应及时更新所存储数据对应的静
态数据标签标识，以保证数据标签标识信息的准确性。
9 日志留存要求
9.1 数据发送方日志要求

9.1.1 数据主体信息

数据发送方应记录数据主体信息，包括数据源主体、数
据接收方和数据传输用途等。其中数据源主体可记录企业统
一社会信用代码，也可直接记录单位名称，或者两者都记录；
数据接收方可记录企业统一社会信用代码，也可直接记录单
位名称，或者两者都记录；数据传输用途可包括数据交易、
委托处理、监管要求等。

9.1.2 数据库审计信息

数据发送方应记录数据库审计信息，包括发送方数据访
问数据库名、表名、字段名、用户名、操作指令等。


16
9.1.3 数据存证信息

数据发送方应记录数据存证信息，包括采样数据和相应
发送的动态标签标识，采样数据留存原则根据统计学中的平
方根采样原则。
注：这种方法能确保较大的元素有更大的被抽中概率，但不会像简单正比
关系那样过度偏向大元素，从而实现相对更均匀的抽样效果；可保证
大数据场景下抽取足够样本，数据量越大、其采样数据比例越小。

9.2 数据接收方日志要求

9.2.1 标签标识验证信息

数据接收方应记录标签验证信息，包括动态标签标识的
接收时间以及核验结果等。

9.2.2 数据存证信息

数据接收方应记录数据存证信息，包括留样数据和接收
的动态标签标识，留样数据留存原则根据统计学中的平方根
采样原则。

10 安全防护要求
10.1 加密传输

数据发送方应采用诸如 HTTPS 的安全传输协议对网络
数据标签标识进行加密传输，具体可参考 GB/T 38636 等。


17
10.2 访问控制

数据接收方应采用访问控制等技术对在内部存储的网
络数据标签标识及相关日志数据进行安全保护，确保仅授权
人员可访问相关信息，具体可参考 GB/T 39205 等。
10.3 防篡改

数据发送方和接收方应采用纠错校验技术或密码技术
确保网络数据标签标识在其生命周期内的完整性和一致性。
10.4 使用环境安全

在网络标签标识的传输、存储等使用环境，应符合国家
网络安全等级保护、数据库管理安全、云计算服务安全等要
求，具体可参考 GB/T 22239、GB/T 20273 和 GB/T 31168 等。

同步传输模式的实现方式示例

本附录在 7.2.2 的示例基础上，给出了网络数据标签标
识同步传输的几种实现方式。
C.1 反向代理实现方式
适用于业务系统使用了 Nginx 等反向代理的情况。此方
式在应用层通过反向代理服务器对 HTTP(S)流量进行中转处
理，利用其头部操作指令（如 add_header, proxy_set_header）
实现标签标识的注入，实现同步传输。
C.2 API 网关实现方式
在系统入口处通过 API 网关组件对所有进出流量进行统
一处理，基于路由规则和过滤（或“插件”）机制实现标签标
识的添加与转发。通过配置 API 网关的策略或编写简单的处
理脚本，可为特定路由的请求或响应添加标签标识，实现同
步传输。
C.3 应用层中间件实现方式
通过在业务应用程序内部嵌入统一的数据标签标识处
理模块。基于拦截器或过滤器机制，在 HTTP 响应返回前，
由中间件组件将数据标签标识嵌入到 HTTP 头部，实现同步
34
传输。
C.4 应用系统实现方式
通过直接修改业务系统服务端每个 API 接口的代码，在
生成响应时，通过调用相应编程语言或框架提供的设置
HTTP 头部的函数（例如，在 Java 中可使用 response.setHeader
方法），将数据标签标识嵌入到 HTTP 响应头部，实现同步
传输。
