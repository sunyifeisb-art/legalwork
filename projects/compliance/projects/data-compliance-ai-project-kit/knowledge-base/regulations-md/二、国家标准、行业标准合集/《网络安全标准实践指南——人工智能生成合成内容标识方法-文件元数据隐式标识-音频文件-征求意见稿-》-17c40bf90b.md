6.1 WAV 格式音频文件的元数据方案

建的具体音视频文件格式。RIFF 文件的基本结构由块（Chunks）组

成，每个块可以包含多个子块（Subchunk），每个子块都有标识符

（Subchunk ID）、大小（Subchunk size）和数据内容（data）。RIFF

文件可以存储多种类型的数据，并在不影响文件解析的情况下添加扩

元数据。

WAV 文件格式的音频文件，采用原生嵌入的元数据方案，具体

实现如下。

a) 存储位置 RIFF 文件中的 Subchunk 中，其中可定义 Subchunk

为 AIGC。

b) 写入方式为：

1) 在 RIFF 文件中插入 Subchunk 块，定义 Subchunk ID 为

AIGC；

2) 在 1）中 ID 为 AIGC 的 Subchunk 块的 Subchunk Data 中，

6.2 MP3 格式音频文件的元数据方案

ID3v2 标签是 MP3 文件中最常用的元数据格式之一，支持更多

的字段和更复杂的数据结构，允许在 MP3 文件的开头存储任意数据。

在 ID3v2 标签中，ID3V2 由 10 个字节的标签头与多个标签帧组成，

标签帧不固定长度。其中定义一个标签帧为 AIGC，在其帧中放置

AIGC 元数据。

对于 MP3 格式的音频文件，采用原生嵌入的元数据方案，具体

实现如下。

a) 存储位置为 ID3v2 的 TXXX 帧；具体来说，是 ID3v2 的 TXXX

（用户自定义文本信息）帧，即 User defined text information

frame。

b) 写入方式为：

1) 在 ID3v2 中，定义一个标签帧为 TXXX_AIGC；具体来说，

是 TXXX 帧，其描述（Description）写入 AIGC；

2) 在标签帧为 AIGC 的帧内容中写入 GB 45438-2025 附录 E

b）所定义的字符串：

{"Label":"value1","ContentProducer":"value2","ProduceID":"

value3","ReservedCode1":"value4","ContentPropagator":"val

6
ue5","PropagateID":"value6","ReservedCode2":"value7"}。

MP3 的实现示例见附录 B。

6.3 OGG 格式音频文件的元数据方案

VORBIS_COMMENT 块是 OGG 格式的音频文件中注释头的标

准元数据格式，一个灵活的元数据存储机制，允许对其添加自定义字

段。该块用于存储人类可读的名称/值对列表。可使用

VORBIS_COMMENT 块放置 AIGC 元数据。

对于 OGG 格式的音频文件，采用原生嵌入的元数据方案，具体

实现如下。

a) 存储位置为 VORBIS_COMMENT 块。

b) 写入方式为：

在 VORBIS_COMMENT 块中，写入 key：AIGC，其 value 值

可 写 入 GB 45438-2025 附 录 E b ） 所 定 义 的 字 符 串 ：

{"Label":"value1","ContentProducer":"value2","ProduceID":"val

ue3","ReservedCode1":"value4","ContentPropagator":"value5","

PropagateID":"value6","ReservedCode2":"value7"}。

OGG 的实现示例见附录 C。

6.4 FLAC 格式音频文件的元数据方案

FLAC 格式是由“fLaC”标记，一个强制性元数据块（称为

STREAMINFO 块）、任意数量的其他元数据块以及音频帧组成。


7

VORBIS_COMMENT 块放置 AIGC 元数据。

对于 FLAC 格式的音频文件，采用原生嵌入的元数据方案，具体

实现如下。

a) 存储位置为 VORBIS_COMMEN 块。

b) 写入方式为：

在 VORBIS_COMMEN 块中，写入 key：AIGC，其 value 值写

{"Label":"value1","ContentProducer":"value2","ProduceID":"val

ue3","ReservedCode1":"value4","ContentPropagator":"value5","

PropagateID":"value6","ReservedCode2":"value7"}。

FLAC 的实现示例见附录 D。

6.5 M4A 格式音频文件的元数据方案

M4A 头文件的基本数据单元是 atom，atom 由 header 和 data 组

成。Atom 由 moov，ftyp，mdat 等常见类型组成。

对于 M4A 格式的音频文件，采用原生嵌入的元数据方案，具体

实现如下。

a) 存储位置为 moov.udta.meta。

b) 写入方式为：

在 moov.udta.meta.keys 中写入 key：AIGC；

8
在 moov.udta.meta.ilst 中写入 value：

{"Label":"value1","ContentProducer":"value2","ProduceID":"val

ue3","ReservedCode1":"value4","ContentPropagator":"value5","

PropagateID":"value6","ReservedCode2":"value7"}。

M4A 的实现示例见附录 E。

XMP 写入实现参考示例


以 python 开发实现为例，示例如下。

a) 安装 exempi，示例代码如下：

pip install python-xmp-toolkit
brew install exempi

b) 写入 XMP 元数据，示例代码如下：

# cac.py
import sys
from libxmp import XMPMeta, XMPFiles

if __name__ == '__main__':
if len(sys.argv) < 2:
print('no argument')
sys.exit()
input_file = sys.argv[1]

# XMP 数据
xmp_data = """<x:xmpmeta xmlns:x="adobe:ns:meta/">
<rdf:RDF xmlns:rdf="http://www.w3.org/1999/02/22-rdf-syntax-ns#"
xmlns:cac="http://ns.cac.gov.cn/ai/1.0/">
<rdf:Description rdf:about="">

<cac:AIGC>{"Label":"value1","ContentProducer":"value2","ProduceID":"value3","ReservedCod
e1":"value4","ContentPropagator":"value5","PropagateID":"value6","ReservedCode2":"value7"}
</cac:AIGC>
</rdf:Description>
</rdf:RDF>
</x:xmpmeta>"""

xmp = XMPMeta()
xmp.parse_from_str(xmp_data)
xmpfile = XMPFiles(file_path=input_file, open_forupdate=True)
xmpfile.put_xmp(xmp)
xmpfile.close_file()

c) 调用，示例代码如下：

python cac.py example.wmv
